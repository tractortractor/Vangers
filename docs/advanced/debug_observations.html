Joystick is now supported

There are 2 things which are controlled by joystick: traction and rudder.

On first joystick setup screen you can choose joystick mode:

  1. NONE - Joystick does not affect traction or rudder. Alternative actions are still active though.
  2. GamePad - Any change of axis position causes maximum change of control value. Basically the same as "Joystick" with max sensitivity.
  3. Joystick - Rudder or traction control values changes proportionally to current axes states. Sensitivity settings affects this mode only.
  4. Steering Wheel - It's supposed to be the same as "Joystick" but without traction axis. Since it's just complete downgrade I decided to not restore this mode but instead leave old broken control code. It's quite fun to try for couple of minutes.

Now there is a second screen in joystick options where you can set which axis controls traction and which axis controls rudder.

To set an axis, click on axis name and move desired axis further than 15/16 of max state. Direction of axis movement determines whether it's inverted.

You can also set sensitivity here.

On first screen of joystick controls you can set key which activates alternative joystick actions.

As long as key is assigned to this activation it cannot be assigned anywhere else.

When this key is pressed positions of assigned axes does not affect traction or rudder but other actions may be triggered. Let's say joystick enters alternative mode in this case.

To assign action on joystick movement in alternative mode go to "customize controls" screen, select control for some action, press and hold "alternative joystick action" button, move one of assigned axes.

DIAGEN_TEST

Loads dialog related variables from "data.lst" if present. Makes all variables non-updatable. See "DATA_UPDATABLE" below.

Prints dialog name, dialog mood (speech mode) and dialog itself in the console.

Creates "Check.LOG" text file where all unused lines are listed.

Creates "diagen_new.text" with all in-game text loaded. This file is almost identical to currently used "diagen.text/diagen_eng.text".

Originally with this test mode data was loaded from "room.lst" file. Since I was not able to figure out how to obtain this file, this feature is now removed from this build. Text data is now loaded from "diagen.text/diagen_eng.text" file.

Controls:

  • "5" - Next dialog line is printed in the console and skipped in game.
  • "6" - Same as previous but done 24 times.
  • "7" - Get list of available questions.
  • "8" - All available questions are asked and answered in the console.
  • "9" - Game process stops and waits for console input (std::cin). Input is saved but not used.
  • "0" - Use last received console input.

Console commands:

  • "GO <escave name>" - Dialog-wise treat current escave as specified for this visit. Names match is case-sensitive. Possible escave names are: Podish, Incubator, VigBoo, Lampasso, Ogorod, ZeePa, B-Zone, Spobs, Threall. "GO Ogorod", for example.
  • "FINISH" - Stop current dialog sequence.
  • "STATUS" - Prints world name, escave name, room name, counselor name and level. Range of values for the level is 0-2. Higher level - more respect for player.
  • "ASK <entry>" - Ask available question. "ASK Agent", for example. See "7" above.
  • "ESTATUS <level>" - Change eleepods' level to specified variable. It's the same level that is printed while invoking STATUS command. "ESTATUS 0", for example.
  • "BSTATUS <level>" - Same as ESTATUS but for beeboorats.
  • "ZSTATUS <level>" - Same as ESTATUS but for zeexs.
  • "SSTATUS <level>" - Added feature. Same as ESTATUS but for spectators.
  • "SAVE" - Save dialog related data in "diagen_saveff.txt" file. Should be viewed with hex editor since values are in raw bytes.
  • "LOAD" - Load dialog related data from "diagen_loadff.txt" file.

Next console commands are dealing with dialog variables.

Originally 2 variable arrays were used by dialog system. First contained global variables and second contained dialog variables. Dialog functions directly queried global variables while those console commands modified dialog variables. Dialog variables were never used, so next commands were completely useless.

In this version dialog variables are used by dialog functions. Each time dialog starts or updates through console commands, dialog variables are updated from global variables. There is also filter array which is used to prevent manually set dialog variables from being updated.

  • "DATA <dialog data index> <new value>" - Set dialog data found by index to specified new value, make this value non-updatable, then perform "DATALIST" action. Both input values must be integers. See "DATALIST".
  • "DATALIST" - Save variables list in "data.lst" file. You can modify those variables with "DATA" and "DATALIST0" commands. visitCounter and comingCounter are not part of data list, so they are not modifiable with mentioned commands. Use "VISIT" instead.
  • "DATALIST0" - Set all variables in data list to 0 and make them non-updatable then perform "DATALIST" action.
  • "DATALIST_READABLE" - Added feature. Save variables in readable form in "readable_data.lst". ID, name and updatability are specified for each variable. Cannot be used as "data.lst" to load variables in game.
  • "DATALIST0_READABLE" - Added feature. Same as "DATALIST0" but saving in readable form in "readable_data.lst" file. Completely useless in practice.
  • "DATA_UPDATABLE <dialog data index> <updatable>" - Added feature. Change updatability of specified variable. Value of updatable dialog variable will be always overwritten by global variable.
  • "DATA_UPDATABLE_ALL <updatable>" - Added feature. Change updatability of all variables.

  • "MAP" - Prints dialogs map in console.

      Legend:

    • space - EMPTY - Unoccupied space.
    • "D" - DORMANT - Currently uninvolved.
    • "U" - USED - Already used.
    • "W" - WAITING - Active, waiting to be checked.
    • "O" - OPENED - Active with access.
    • "E" - EXPLODED - Involved.
    • skipped while mapped - HALFLIFE - Waiting for end (repeatable = 1).
  • "VISIT <escave ID> <visitCounter> <comingCounter>" - Change visitCounter and comingCounter of specified escave to specified values. Escave ID is first symbol of name, case-sensitive. See "GO" above for full list of names. visitCounter and comingCounter values are never actually used.

zRECORDER_ENABLED

Not defined anywhere.


IMPORTANT! Backup saves and options before playing replays. Save file will be overwritten during playing of replay if replay creator saved the game during recording. Other settings may be changed in similar way. If your build is compiled with ENTRIES_CONTROL and MSG_OUT macros then replay can also change "*.prm" files. Saves are stored in "savegame" folder. Options are stored in "game.dat" and "controls.dat" files.


Records player's action if command line argument "w" or "W" is specified. File where to write must be specified right after this argument. Example: "/wfile_where_to_save", "/Wother_file_where_to_save".

Loads player's action if command line argument "p" or "P" is specified. File where to read from must be specified right after this argument. Example: "/pfile_where_to_read", "/Pother_file_where_to_read". Any control action which is not mouse moving, releasing keyboard button or releasing mouse button will stop playback.


Replays should be played with same build of vangers which were used for recording.

It is unknown if replays are cross-platform, so it's a good idea to specify your OS while posting them.

Options which are ok to modify while recording: all camera options, autorun, deformable landscape. Modification of other options may mess up recording/playing.


RNG seeds, controls and some options are stored in replay file.

Each controls event takes 8 bytes to record. For example event is created each time button is pressed or released.

While recording with joystick, replay size will be several times bigger, since any stick move creates new event.

While "GamePad" mode is on the game will record joystick movement events only when axis changes direction.

Mouse movements are not recorded so game interface does not react to them while recording.

While recording it's possible to steal counselor's items through walls. This was made to deal with absence of mouse movements while recording and to make replays more interesting to watch.
